  • +Ad Jesum Per Mariam+

    Happy Holidays, Tijl Kindt!

    Well it depends on what you define as "unfinished." I use something similar to AF4R, which differentiates new/old, recurrent, and unfinished tasks.

    Now please note I said "TASKS." Projects are, by definition, a collection of tasks towards a certain goal, and this definition guides me on how to treat my projects.

    If I am going to make a new project, I make a new file and enter the name into the AF list as "New."

    If I have worked a project >>to the completion of one or more of its tasks BUT NOT to the completion of the project itself,<< I re-enter the project name into the AF list as "Recurrent."

    If I worked on a project >>but did not finish the current task in that project,<< I re-enter the name of the project AND the task that was not finished into the AF list as "Unfinished."

    If I finished >>the project itself<< I delete it from my AF list.

    Do these all make sense?

  • Tijl Kindt asked ...
    << Do you put "Work on project X" in the Unfinished section and then single tasks of a project in Old/New? Or everything on project X in Unfinished until the whole project is finished? >>

    I tend to combine project/subtask as one task in DWM. I believe you can do the same in AF4R.

    So I have tasks in the form "Work on project X (one or two next actions)". When I remember a new next actions for project X, I either add a new subtask in Project X reference check list or just add into DWM list a taks in the form "Do one next action for project X". When the latter stand out, I either do it immediately or consolidate that subtask into project X check list.

    I also have a task in DWM to "Review projects", where I'll review my projects list and make sure there's at least one task in DWM for each active project.

    If I didn't finish a task "as written", I'll Re-enter (1W ahead) incomplete task (AF4R = unfinished). If I did finish it, I'll Enter (1M ahead) a new task (AF4R = old/new) for that Project X with relevant next action.

    AF/DWM is so very flexible in defining task size, that I think we should really exploit it. ;-)
